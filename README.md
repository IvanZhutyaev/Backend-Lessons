# BACKEND

## FRONTEND vs BACKEND

**Frontend** - клиентская часть продукта. В случае с сайтом ее формирует и выводит на экран браузер.

**Backend** - внутренняя часть продукта, которая находится на сервере и скрыта от пользователей.

## HTTP протокол

**HTTP** - протокол передачи гипертекста. Это набор правил, по которым данные в интернете передаются между разными источниками, обычно между компьютерами и серверами. Интернет-протокол HTTP - это шаблон, по которому формируется запрос на передачу данных, а затем передаются интернет-страницы, видео, аудио и текст.

### Структура HTTP-сообщения

Структура HTTP-сообщения всегда одинакова:

1. **Стартовая строка** - определяется адрес, по которому отправляется запрос, и тип сообщения. Указывается метод, который определяет действия при получении этого сообщения. Это может быть чтение данных, их отправка, изменение и удаление.
2. **Заголовки** - прописаны определённые параметры сообщения.
3. **Тело запроса** - текст сообщения, данные, которые передаются.

### Основные методы HTTP

| Метод | Описание |
|-------|----------|
| **GET** | Используется для запроса данных с сервера (получает, но не изменяет; сохраняется в истории) |
| **POST** | Отправка данных на сервер для создания или обновления ресурса (не сохраняется в истории; в ответ сервер выполнит фоновый процесс или отчитается) |
| **PUT** | Отправка данных на сервер для создания или обновления (не требует от сервера ответа) |
| **PATCH** | Используется для частичного изменения ресурса (инструкция по обновлению нужной части) |
| **DELETE** | Используется для удаления запрашиваемого ресурса (полностью удаляет ненужную информацию) |

### Принцип работы

Работа HTTP-протокола максимально проста - клиент передаёт запрос, сервер формирует ответ и передаёт обратно.

## REST

**REST** - *Representational State Transfer* (передача состояния представления). REST определяет, как компоненты должны взаимодействовать друг с другом (отправка запроса - клиент, обработка - сервер).

### Принципы REST

1. **Приведение архитектуры к модели клиент-сервер** - нужно разграничивать обязанности клиента и сервера.
2. **Отсутствие состояния** - запрос должен иметь ВСЮ необходимую информацию.
3. **Кэширование** - клиенты могут выполнять кэширование ответов сервера.
4. **Единообразие интерфейса** - унифицированный интерфейс.
5. **Слои** - применение промежуточных серверов для балансировки нагрузки и распределения кэширования (прокси-сервер, микро-сервис, балансировщик).

### Методы REST API

**CRUD** - Create, Read, Update, Delete.

### Endpoint

**Endpoint** - конечная точка веб-сервиса, предоставляющая доступ к определённому ресурсу или набору ресурсов.

## SPRING BOOT

**Spring Boot** - фреймворк для разработки бека на Java.

### Что такое фреймворк?

**Фреймворк** - заготовка, готовая модель в программировании для быстрой разработки, на основе которой можно дописать собственный код. Он задаёт структуру, определяет правила и предоставляет необходимый набор инструментов для создания проекта.

### Web App

**Web App** - программа с определённым набором функционала, использующая в качестве клиента браузер.

### Составляющие Spring Framework

| Модуль | Описание |
|--------|----------|
| **Spring Core Container** | Модуль с базовыми функциями фреймворка (внедрение зависимостей и управление жизненным циклом бинов) |
| **Spring AOP** | Модуль, который позволяет реализовывать аспектно-ориентированное программирование в приложениях |
| **Spring Data Access/Integration** | Набор модулей, обеспечивающих доступ к данным из различных источников, включая реляционные БД |
| **Spring WEB** | Модули, предназначенные для разработки веб-приложений и веб-сервисов |
| **Spring Test** | Модуль для тестирования |
| **Spring Security** | Модуль для обеспечения безопасности в приложениях |
| **Spring Transactions** | Модуль поддержки транзакций в web-приложениях |

### Spring Core (Bean)

**Bean Spring** - объект, управляемый контейнером Spring. Он представляет собой класс, который Spring управляет и внедряет в другие классы в зависимости от их конфигурации.

### Жизненный цикл Spring приложения

```
Запуск Spring приложения 
    ↓
Запуск Spring контейнера 
    ↓
Создание объекта бина 
    ↓
Внедрение зависимостей в бин 
    ↓
Вызов указанного init-method 
    ↓
Бин готов к использованию 
    ↓
Работа бина 
    ↓
Вызов указанного destroy-method 
    ↓
Остановка Spring приложения
```

### Инверсия управления(Inversion of Control)
**Состоит в том, что большую часть объектов создаём не мы, а Spring. Мы лишь конфигурируем классы**

### Dependency injection
**Внедрение зависимости - это и есть инициализация полей бинов другими бинами
Ведь помимо создания объектов, Spring-контейнер внедряет эти объекты в другие объекты, т.е. делает их полями других объектов**

#### Аннотация @Component
**Помечаем ей класс, если хотим, чтобы Spring Framework создал бин этого класса**


@PostConstruct - после конструирования
@PreConstruct - перед уничтожением
Может быть любой модификатор доступа
Тип возвращаемого значения может быть любой
Название метода может быть любым, а также они не должны принимать на вход какие-либо аргументы

@Configuration - помечает класс для создания конфигуации

MVC - архитектурный паттерн, который делит программу на 3 независимых компонента
1)Контроллер
Обрабатывает запросы от пользовател
Обменивается данными с моделью
Показывает пользователю правильное представление
Переадресовывает пользователя на другие страницы

2)Модель(Сущность)
Хранит в себе данные
Взаимодействует в бд для получения данных
Отдаёт данные контроллеру

3)Представление
Получает данные от контроллера и отображает их в браузере
Для динамического отображения данных используются шаблонизаторы


Слои приложения(Уровни ответственности)
Контроллеры  - обработка HTTP-запросов
Сервисы - бизнес-логика
Репозитории взаимодействие с бд
Модели  - сущности бд/DTO
Конфигурация - конфигурационный код
